# AI 保留/删除建议模块 — 完整技术文档

## 1. 模块概述

### 1.1 模块目标

本模块是整个照片整理工具的核心功能。目标是学习用户的个人审美偏好，对大量未标注的下载图片给出"建议保留"或"建议删除"的二元判断，从而将图片筛选工作从"完全人工逐张判断"转变为"AI 预处理 + 人工复核修正"。

### 1.2 核心设计原则

1. **AI 只有建议权，没有执行权** — 所有删除操作必须用户确认后才执行
2. **输出始终为二元结果** — 保留或删除，附带置信度分数，不做多级分类
3. **持续学习** — 用户的每次修正都是新的训练数据，模型随使用越来越准
4. **所有照片一个模型** — 不按内容类型拆分，单一全局模型学习用户的统一审美偏好
5. **完全离线** — 所有计算在本地完成，不依赖任何网络服务

### 1.3 适用场景

- PT 下载的写真集、图片包
- 批量网络下载图片
- 任何需要从大量图片中筛选"我喜欢的"的场景

### 1.4 不适用场景

- 视频筛选（不参与）
- RAW 文件（不参与）
- 日常手机照片整理（这类场景通常不需要 AI 筛选，用重复检测和时间整理即可）

### 2.2 两阶段解耦设计

本模块的核心设计是将"理解图片"和"学习偏好"解耦为两个独立阶段：

```
阶段 A: 特征提取（OpenCLIP，冻结不训练）
    图片 → OpenCLIP ViT-H/14 → 1024 维特征向量 → 缓存到数据库

阶段 B: 偏好学习（MLP，用户数据训练）
    特征向量 + 用户标注 → 训练 MLP → 输出保留概率
```

**为什么要解耦**：

| 操作 | 阶段 A（特征提取） | 阶段 B（偏好学习） |
|------|------------------|------------------|
| 何时运行 | 新图片进入时，一次性 | 用户标注后，可反复 |
| 计算成本 | 较高（GPU，~50 img/s） | 极低（CPU 即可，秒级） |
| 是否需要 GPU | 是 | 否（但用 GPU 更快） |
| 输出 | 1024 维浮点数组 | 保留概率 0.0~1.0 |
| 是否缓存 | 是，永久缓存 | 不需要缓存 |

用户修正标注后重新训练，只需跑阶段 B（秒级），不需要重新提取特征。

## 3. 模型选型

### 3.1 特征提取器：OpenCLIP ViT-H/14

#### 选型决策

| 候选方案 | 淘汰原因 |
|---------|---------|
| EfficientNet-B0 (ImageNet) | ImageNet 特征为物体识别设计，不含审美维度，准确率低 5-10% |
| ResNet-50 (ImageNet) | 同上 |
| DINOv2 (Meta) | 纯视觉自监督，不理解"美丑"等语义概念，不适合审美任务 |
| OpenAI CLIP 原版 | 性能与 OpenCLIP 相当，但开源程度和社区支持不如 OpenCLIP |
| OpenCLIP ViT-B/32 | 性能可用但特征维度低（512），小数据量时准确率差 3-5% |
| OpenCLIP ViT-L/14 | 良好选择，但 16GB VRAM 完全可以跑更大的 H/14 |
| OpenCLIP ViT-bigG/14 | 收益递减（仅多 1%），速度慢一倍以上 |

#### 最终选择

```
模型：OpenCLIP ViT-H/14
预训练权重：laion2b_s32b_b82k
特征维度：1024
VRAM 占用：~4 GB
推理速度：~50 img/s（RTX 4070 Ti Super）
```

#### 选择 OpenCLIP 的核心原因

OpenCLIP 在 20 亿图文对上训练，其特征空间天然编码了审美维度：

- **构图**：见过"good composition"、"rule of thirds"等描述
- **光线**：见过"dramatic lighting"、"soft light"、"golden hour"等描述
- **色调**：见过"warm tones"、"vibrant colors"、"moody"等描述
- **风格**：见过"cinematic"、"minimalist"、"vintage"等描述
- **内容**：见过"portrait"、"landscape"、"still life"等描述

用户不需要教 AI"什么是构图好"——OpenCLIP 已经知道。用户只需要通过标注告诉 AI：**"在所有这些维度中，我个人偏好的组合是什么"**。

### 3.2 分类器：MLP 二分类器

#### 架构

```
输入层:  1024 维（OpenCLIP ViT-H/14 输出）
    ↓
隐藏层:  Linear(1024, 256) + ReLU + Dropout(0.3)
    ↓
输出层:  Linear(256, 1) + Sigmoid
    ↓
输出:    0.0 ~ 1.0（保留概率）
```

- 参数量：~262,000（约 1 MB），备份 15 个版本仅占 ~15 MB
- Dropout 0.3：在小数据量（200-500 张）时防止过拟合

#### 损失函数：Focal Loss

```
Focal Loss = -(1 - pt)^γ * log(pt)

其中 pt = 模型预测正确的概率
γ = 2（聚焦参数）
```

作用：自动降低"模型已经判断对了的简单样本"的权重，让训练更关注"判断困难的边界样本"。无论保留:删除的比例是多少，Focal Loss 都能稳定训练。

#### 训练配置

```
优化器：Adam，学习率 1e-3
批大小：64
轮次：50 epoch（带早停，验证损失 5 轮不下降则停止）
数据划分：80% 训练 / 20% 验证
均衡采样：每个 epoch 中保留和删除样本各取等量（上采样少数类）
设备：CUDA（GPU）
```

#### 输出解释

```
概率 > 0.5  →  建议保留
概率 < 0.5  →  建议删除

置信度 = |概率 - 0.5| × 2，归一化到 0.0 ~ 1.0
    概率 = 0.95 → 置信度 0.9（非常确定保留）
    概率 = 0.52 → 置信度 0.04（几乎不确定）
    概率 = 0.10 → 置信度 0.8（非常确定删除）
```

## 4. 数据设计

### 4.1 数据库表

#### files 表（扩展字段）

```sql
-- 在已有的 files 表中，AI 模块使用以下字段：
clip_feature    BLOB        -- OpenCLIP ViT-H/14 特征向量
                            -- 存储格式：1024 × float32 = 4096 字节/张
                            -- 仅图片文件有值，视频和其他文件为 NULL
```

#### ai_labels 表

```sql
ai_labels (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    file_id         INTEGER NOT NULL REFERENCES files(id),
    task_id         INTEGER NOT NULL REFERENCES tasks(id),

    -- 用户标注
    user_label      TEXT,           -- 'keep' / 'delete' / NULL（未标注）
    labeled_at      DATETIME,       -- 用户标注时间

    -- AI 预测
    ai_prediction   TEXT,           -- 'keep' / 'delete' / NULL（未预测）
    ai_confidence   REAL,           -- 0.0 ~ 1.0 置信度
    ai_raw_score    REAL,           -- 0.0 ~ 1.0 原始保留概率
    predicted_at    DATETIME,       -- 预测时间
    model_version   INTEGER,        -- 使用的模型版本号

    -- 标记来源
    is_training_data BOOLEAN DEFAULT FALSE,  -- 是否已被纳入训练集
    label_source    TEXT,           -- 'manual'（手动标注）/ 'correction'（修正 AI 结果）

    UNIQUE(file_id, task_id)
)
```

#### ai_model_versions 表

```sql
ai_model_versions (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    version         INTEGER NOT NULL UNIQUE,    -- 版本号，递增
    model_path      TEXT NOT NULL,              -- 模型文件路径

    -- 训练信息
    training_samples    INTEGER,    -- 训练样本总数
    keep_samples        INTEGER,    -- 其中"保留"样本数
    delete_samples      INTEGER,    -- 其中"删除"样本数

    -- 评估指标
    val_accuracy    REAL,           -- 验证集准确率
    val_precision   REAL,           -- 验证集精确率（保留类）
    val_recall      REAL,           -- 验证集召回率（保留类）
    val_f1          REAL,           -- 验证集 F1 分数

    -- 元数据
    training_time_sec   REAL,       -- 训练耗时（秒）
    created_at          DATETIME,
    is_current          BOOLEAN DEFAULT FALSE   -- 是否为当前使用的版本
)
```

#### ai_training_config 表

```sql
ai_training_config (
    id              INTEGER PRIMARY KEY,
    min_samples     INTEGER DEFAULT 200,    -- 冷启动最低标注数
    model_backbone  TEXT DEFAULT 'ViT-H-14',
    pretrained      TEXT DEFAULT 'laion2b_s32b_b82k',
    feature_dim     INTEGER DEFAULT 1024,
    hidden_dim      INTEGER DEFAULT 256,
    dropout         REAL DEFAULT 0.3,
    learning_rate   REAL DEFAULT 0.001,
    batch_size      INTEGER DEFAULT 64,
    max_epochs      INTEGER DEFAULT 50,
    early_stop_patience INTEGER DEFAULT 5,
    focal_gamma     REAL DEFAULT 2.0,
    max_model_backups   INTEGER DEFAULT 15
)
```

```python
import numpy as np
import struct
```

## 5. 完整工作流程

### 5.1 流程总览

```
┌──────────────────────────────────────────────────────────────┐
│                        用户视角流程                            │
│                                                              │
│  选择文件夹 → 扫描 → 提取特征 → 冷启动标注 → 训练 → AI 建议    │
│                                      ↑                 │     │
│                                      └── 修正 ←────────┘     │

│                                       确认执行删除             │

```

### 5.2 阶段一：特征提取

#### 触发时机

当用户在某个任务中进入 AI 模块时，检查该任务中所有图片文件是否已提取特征。对未提取特征的图片启动批量提取。

#### 流程

```
1. 查询 files 表，筛选 task_id = 当前任务 AND file_type = 'image' AND clip_feature IS NULL
2. 加载 OpenCLIP ViT-H/14 模型到 GPU
3. 按 batch_size=32 批量处理：
   a. 从磁盘读取图片
   b. 预处理（resize + normalize）
   c. 模型前向推理 → 1024 维特征向量
   d. L2 归一化
   e. 写入 files 表的 clip_feature 字段
4. 通过 WebSocket 推送提取进度（已处理/总数）
5. 提取完成后释放 GPU 显存（卸载 OpenCLIP 模型）
```

#### 异常处理

| 场景 | 处理方式 |
|------|---------|
| 图片文件损坏无法读取 | 跳过该文件，标记 clip_feature = NULL，在 UI 中提示 |
| 图片格式不支持 | 跳过，提示用户 |
| GPU 显存不足 | 自动降低 batch_size 重试（32→16→8→1） |
| 提取过程中断（关闭软件） | 下次进入时只处理 clip_feature IS NULL 的文件，已完成的不重复 |

#### 性能预期

| 图片数量 | 提取耗时 | VRAM 占用 |
|---------|---------|----------|
| 1,000 | ~20 秒 | ~4.5 GB |
| 5,000 | ~100 秒 | ~4.5 GB |
| 10,000 | ~3.5 分钟 | ~4.5 GB |
| 50,000 | ~17 分钟 | ~4.5 GB |

### 5.3 阶段二：冷启动标注

#### 触发条件

当前全局标注数据（所有任务累计）不足 200 条时，进入冷启动模式。

#### 冷启动 UI 状态

```
┌─────────────────────────────────────────────────────┐
│  ⚠ AI 模型尚未就绪                                    │
│                                                     │
│  需要标注至少 200 张图片才能开始训练。                    │
│  当前已标注：47 / 200                                 │
│                                                     │
│  ████████░░░░░░░░░░░░  23.5%                        │
│                                                     │
│  请在下方浏览图片并标记"保留"或"删除"。                   │
│  标注达到 200 张后，即可训练 AI 模型。                   │
└─────────────────────────────────────────────────────┘
```

#### 标注方式

用户直接在当前任务的图片列表中逐张或批量标注保留/删除。不需要 AI 帮忙挑选——冷启动阶段没有模型，无法判断哪些图更值得标注。

### 5.4 阶段三：模型训练

#### 触发条件

- 冷启动完成（标注 ≥ 200 张）后，用户点击"训练模型"按钮
- 后续任何时候，用户修正了一些标注后，可再次点击"重新训练"

#### 训练流程

```
1. 收集训练数据
   - 从 ai_labels 表查询所有 user_label IS NOT NULL 的记录
   - 关联 files 表获取对应的 clip_feature
   - 最终得到：[(feature_vector, label), ...] 数据集

2. 备份当前模型
   - 如果存在当前模型，复制到 history/ 目录
   - 检查历史版本数量，超过 15 个则删除最早的

3. 数据准备
   - 按 80/20 划分训练集/验证集（分层采样，保持比例）
   - 对训练集做均衡采样（上采样少数类，使保留和删除数量相等）

4. 训练
   - 初始化 MLP 模型 (1024 → 256 → 1)
   - Focal Loss (γ=2.0)
   - Adam 优化器 (lr=1e-3)
   - 逐 epoch 训练，每个 epoch 后在验证集上评估
   - 早停：验证损失 5 轮不下降则停止
   - 保存验证准确率最高的模型权重

5. 评估
   - 在验证集上计算：准确率、精确率、召回率、F1
   - 将评估结果写入 ai_model_versions 表

6. 部署
   - 将最佳模型保存为 current/classifier.pt
   - 更新 ai_model_versions 表的 is_current 字段

7. 自动推理
   - 训练完成后，自动对当前任务中所有未预测的图片执行推理
```

#### 异常处理

| 场景 | 处理方式 |
|------|---------|
| 标注全部为同一类（都是保留或都是删除） | 拒绝训练，提示用户需要两类标注 |

| 验证集过小（标注 < 20 张的那一类） | 警告用户数据不足，建议多标注少数类 |

### 5.5 阶段四：AI 建议与用户审查

#### 推理流程

```
1. 加载当前模型 current/classifier.pt
2. 从 files 表批量读取 clip_feature（已缓存）
3. 批量通过 MLP → 输出保留概率
4. 计算置信度
5. 结果写入 ai_labels 表
6. 通过 WebSocket 通知前端刷新
```

推理速度：特征已缓存时 > 10,000 img/s（纯 MLP 计算，瞬间完成）

#### 审查交互（核心 UI）

详见下方第 6 节 UI/UX 设计。

### 5.6 阶段五：确认执行删除

#### 流程

```
1. 用户在 AI 模块中完成所有审查和修正
2. 用户点击"执行删除"按钮
3. 弹出确认对话框，展示：
   - 将删除的文件总数

6. 每个文件操作记录到 operation_logs 表

```

| 筛选维度 | 选项 |

| 置信度 | 全部 / 高（>80%）/ 中（30-80%）/ 低（<30%） |

#### 排序选项

| 排序方式 | 说明 |
|---------|------|
| 置信度（默认） | 最不确定的排在最前面，方便用户优先审查边界案例 |
| 文件名 | 字母顺序 |
| 文件大小 | 从大到小或从小到大 |
| AI 保留概率 | 从高到低或从低到高 |

#### 推荐审查流程

```
1. 先筛选"置信度 < 30%"的图片（AI 最不确定的）
   → 这些是最需要人工判断的，逐张检查

2. 再筛选"AI 建议保留"的图片，快速扫一遍
   → 确认没有明显误判的

3. 再筛选"AI 建议删除 + 置信度 > 80%"的图片，抽查几张
   → 确认 AI 的高置信度删除判断是否合理

4. 对于 AI 建议删除 + 置信度 > 90% 的图片
   → 通常可以直接接受，不需要逐张检查
```

## 7. 场景与边界情况

### 7.1 正常使用场景

| 场景 | 流程 |
|------|------|
| 首次使用（无模型） | 进入 AI 模块 → 提取特征 → 标注 200 张 → 训练 → 审查建议 → 修正 → 确认删除 |
| 日常使用（有模型） | 进入新任务的 AI 模块 → 提取特征 → 自动推理 → 审查建议 → 修正 → 可选重训练 → 确认删除 |
| 累积改进 | 多批任务后，修正数据累积 → 重新训练 → 模型越来越准 |

### 7.2 边界情况

| 场景 | 处理方式 |
|------|---------|
| 图片文件损坏 | 跳过特征提取，在列表中标记为"无法处理"，不参与 AI 判断 |
| 图片格式不支持 | 同上 |
| 任务中只有少量图片（<10 张） | 正常展示，但提示"图片数量较少，建议直接手动标注" |
| 所有标注都是同一类 | 拒绝训练，提示需要两类标注数据 |
| 标注数据极度不均衡（如 190:10） | 允许训练，但警告"删除/保留样本过少，可能影响准确率" |
| 用户回滚模型后重新推理 | 清除当前任务的旧 AI 预测，用回滚的模型重新推理 |
| 用户在多个任务中标注 | 所有任务的标注数据合并训练，模型是全局的 |
| 模型训练中关闭软件 | 训练结果丢失，旧模型不受影响，下次可重新训练 |
| 特征提取中关闭软件 | 已提取的特征保留在数据库，下次只处理剩余文件 |
| 同一张图片在不同任务中出现 | 各任务独立标注和预测，但训练时合并所有标注 |

### 7.3 性能边界

| 场景 | 预期表现 |
|------|---------|
| 1,000 张图片 | 特征提取 ~20 秒，推理 <1 秒 |
| 10,000 张图片 | 特征提取 ~3.5 分钟，推理 <1 秒 |
| 50,000 张图片 | 特征提取 ~17 分钟，推理 ~1 秒 |
| 100,000 张图片 | 特征提取 ~35 分钟，推理 ~1 秒 |
| 训练 200 样本 | ~10 秒 |
| 训练 2000 样本 | ~30 秒 |
| 训练 10000 样本 | ~2 分钟 |

### 9.2 OpenCLIP 加载/卸载策略

OpenCLIP ViT-H/14 占用 ~4GB VRAM，不应常驻 GPU。策略：

```
特征提取时：
    1. 加载 OpenCLIP 到 GPU
    2. 批量处理所有图片
    3. 处理完毕后卸载 OpenCLIP，释放显存

训练时：
    不需要 OpenCLIP（特征已缓存），直接训练 MLP

推理时：
    不需要 OpenCLIP（特征已缓存），直接跑 MLP
```

MLP 分类器仅 ~1MB，可常驻内存，不占用显著资源。

## 10. 精度预期与优化路径

### 10.1 各阶段精度预期

| 阶段 | 标注量 | 预期准确率 | 用户需手动审查比例 |
|------|-------|----------|----------------|
| 冷启动 | 200 | 78-85% | ~20% 需要修正 |
| 初期 | 500 | 82-88% | ~15% 需要修正 |
| 中期 | 1000 | 85-92% | ~10% 需要修正 |
| 成熟 | 2000+ | 88-95% | ~5% 需要修正 |

### 10.2 如果精度不达标

如果实际使用中精度低于预期，按以下顺序排查和优化：

| 优先级 | 优化手段 | 适用场景 | 实施复杂度 |
|--------|---------|---------|-----------|
| 1 | 增加标注数据 | 标注 <500 且不满意 | 零（用户继续标注即可） |
| 2 | 检查标注质量 | 用户可能有不一致的标注 | 低（提供标注一致性检查工具） |
| 3 | 调整 MLP 结构 | 256 隐藏层可能不够 | 低（改配置重训练） |
| 4 | 添加 LAION 美学分数 | 边界案例多 | 中（额外特征拼接） |
| 5 | LoRA 微调 OpenCLIP | 标注 >2000 且准确率停滞 | 中高 |
